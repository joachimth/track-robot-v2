# CLAUDE.md  Development Guidelines for AI Assistants

This file provides instructions and conventions for AI assistants (like Claude) working on this repository. It ensures consistency, maintainability, and adherence to project goals.

## Project Goals

### Primary Goals
1. **Production-ready tracked robot firmware** for ESP32-WROOM-32
2. **Multiple control methods**: PS4 DualShock 4 (primary, via Bluepad32 + BTstack), Serial (UART), HTTP (Wi-Fi)
3. **Safety-first design**: E-stop, arming, failsafe timeout, initial disarmed state
4. **High-quality motor control**: 20 kHz PWM, smooth differential drive, configurable parameters
5. **Full automation**: CI/CD builds, releases, and deploys web flasher with zero manual steps
6. **Comprehensive documentation**: Every decision explained, every interface documented

### Non-Goals
- ✗ Not a general-purpose robot platform (scoped to tracked robots)
- ✗ Not supporting other MCUs (ESP32 only)
- ✗ Not supporting other motor drivers (BTS7960 / IBT-2 only)
- ✗ Not a learning framework (production code, not tutorial)

### Naming Conventions

- **Files**: `snake_case.c`, `snake_case.h`
- **Directories**: `lowercase` (no underscores)
- **Components**: `lowercase` (e.g., `control`, `motor`, `safety`)
- **Functions**: `component_action()` (e.g., `motor_bts7960_set_speed()`)
- **Types**: `component_type_t` (e.g., `control_frame_t`)
- **Defines**: `COMPONENT_CONSTANT` (e.g., `MOTOR_PWM_FREQ_HZ`)
- **Kconfig**: `CONFIG_ROBOT_*` prefix for all project options

## Coding Conventions

### C Code Style

- **Standard**: C11
- **Indentation**: 4 spaces (no tabs)
- **Braces**: K&R style (opening brace on same line)
- **Line length**: 100 characters max
- **Comments**: `//` for single-line, `/* */` for multi-line
- **Headers**: Include guards using `#pragma once`

#### Example:

```c
#pragma once

#include <stdint.h>
#include "esp_err.h"

/**
 * @brief Initialize the BTS7960 motor driver
 *
 * @param config Motor configuration structure
 * @return esp_err_t ESP_OK on success
 */
esp_err_t motor_bts7960_init(const motor_config_t *config);

/**
 * @brief Set motor speed (-1.0 to +1.0)
 *
 * @param motor_id Motor identifier (MOTOR_LEFT or MOTOR_RIGHT)
 * @param speed Normalized speed value
 * @return esp_err_t ESP_OK on success
 */
esp_err_t motor_bts7960_set_speed(uint8_t motor_id, float speed);
```

### Logging

Use ESP-IDF logging macros:

```c
#include "esp_log.h"

static const char *TAG = "motor_bts7960";

ESP_LOGI(TAG, "Motor initialized: PWM freq=%d Hz", freq);
ESP_LOGW(TAG, "Motor speed clamped: %.2f -> %.2f", raw, clamped);
ESP_LOGE(TAG, "Failed to initialize PWM: %s", esp_err_to_name(ret));
```

**Log levels**:
- `ESP_LOGI`: Normal operation (startup, state changes)
- `ESP_LOGD`: Debug info (stick values, calculations)
- `ESP_LOGW`: Warnings (clamping, timeouts)
- `ESP_LOGE`: Errors (initialization failures, invalid params)

### Error Handling

- **Return `esp_err_t`** for all functions that can fail
- **Check all return values**: Never ignore errors
- **Fail fast**: Return early on error, don't continue with invalid state
- **Log errors**: Always log at `ESP_LOGE` level before returning error

```c
esp_err_t ret = some_function();
if (ret != ESP_OK) {
    ESP_LOGE(TAG, "Failed to do something: %s", esp_err_to_name(ret));
    return ret;
}
```

## Architecture Contracts

### Control Frame (Interface)

All control sources MUST produce a normalized control frame:

```c
typedef struct {
    float throttle;      // -1.0 (full reverse) to +1.0 (full forward)
    float steering;      // -1.0 (full left) to +1.0 (full right)
    bool estop;          // Emergency stop command
    bool arm;            // Arming command
    bool slow_mode;      // Slow mode toggle
    uint32_t timestamp;  // ESP32 tick count (for timeout detection)
} control_frame_t;
```

**Rules**:
- `throttle` and `steering` MUST be clamped to [-1.0, +1.0]
- `timestamp` MUST be updated on every new input
- Controller modules call `control_manager_submit(source_id, &frame)`

### Adding a New Control Source

1. **Create module**: `firmware/components/control/controller_myname.c`
2. **Implement interface**:
   - `esp_err_t controller_myname_init(void)`
   - `void controller_myname_task(void *arg)` (FreeRTOS task)
   - Produce `control_frame_t` and call `control_manager_submit()`
3. **Register in Kconfig**: Add enable option in `main/Kconfig.projbuild`
4. **Initialize in main.c**: Call `controller_myname_init()` if enabled
5. **Document**: Add new doc file `docs/myname-protocol.md`
6. **Update**: PROGRESS.md, README.md, architecture.md

### NVS Config Contract

Drive parameters are stored in NVS namespace `robot_cfg`:

| Key | Type | Range | Kconfig fallback |
|-----|------|-------|-----------------|
| `deadzone` | int32 (%) | 0–20 | `CONFIG_ROBOT_DRIVE_DEADZONE` |
| `expo` | int32 (%) | 0–100 | `CONFIG_ROBOT_DRIVE_EXPO` |
| `max_speed` | int32 (%) | 10–100 | `CONFIG_ROBOT_DRIVE_MAX_SPEED` |
| `slow_factor` | int32 (%) | 10–100 | `CONFIG_ROBOT_DRIVE_SLOW_MODE_FACTOR` |

`main.c` reads NVS at boot (before `mixer_diffdrive_init`) and overrides Kconfig defaults.
`controller_http.c` exposes `GET /config` and `POST /config` to read/write these values.
Changes take effect after reboot.

WiFi STA credentials are stored separately in namespace `wifi_cfg` (keys: `ssid`, `password`).

### Motor Control Contract

Motor module accepts normalized speed:

```c
esp_err_t motor_set_speeds(float left, float right);
```

- Inputs: -1.0 (full reverse) to +1.0 (full forward)
- Module handles: PWM generation, direction control, enable pins, ramping
- Safety: Stops motors if `motor_emergency_stop()` called

## Bluetooth / Gamepad Backend

### Bluepad32 + BTstack

The PS4 controller support is implemented via **Bluepad32** (a third-party
gamepad library) running on top of **BTstack** (a portable BT stack).

**Why Bluepad32?**
- Supports PS4, PS5, Xbox, Nintendo Switch controllers uniformly
- More stable than ESP-IDF's native `esp_hidh` for gamepad use cases
- Abstracts HID report parsing — no manual byte-offset work needed

**Key components**:
- `firmware/components/ps4/` — Bluepad32 platform adapter + `ps4_gamepad_t` abstraction
- `firmware/components/cmd_nvs/` — Stub required by Bluepad32's build system
- `firmware/components/cmd_system/` — Stub required by Bluepad32's build system

**sdkconfig.defaults** disables Bluedroid (`CONFIG_BT_BLUEDROID_ENABLED=n`) and
enables BTstack controller-only mode (`CONFIG_BT_CONTROLLER_ONLY=y`).

**DO NOT** re-enable Bluedroid or `esp_hidh` — they conflict with BTstack.

## Configuration Management

### Kconfig

All user-configurable options MUST be in Kconfig:

- **File**: `firmware/main/Kconfig.projbuild`
- **Prefix**: `CONFIG_ROBOT_*`
- **Sections**: Motor Pins, Motor Control, Differential Drive, Control Sources, WiFi, Safety

#### Example:

```kconfig
menu "Robot Configuration"

    menu "Motor Pins"
        config ROBOT_MOTOR_LEFT_RPWM
            int "Left motor RPWM pin"
            default 27
            range 0 39
    endmenu

    menu "Motor Control"
        config ROBOT_MOTOR_PWM_FREQ_HZ
            int "PWM frequency (Hz)"
            default 20000
            range 1000 40000
            help
                Higher frequency = smoother control, less audible noise.
                BTS7960 supports up to ~25kHz. Default 20kHz is optimal.
    endmenu

endmenu
```

### Defaults

**File**: `firmware/sdkconfig.defaults`

Contains production defaults for:
- ESP32-WROOM-32 target (4 MB flash, no PSRAM)
- Bluetooth Classic (BR/EDR + BLE) via BTstack (Bluedroid disabled)
- Bluepad32 options (`CONFIG_BLUEPAD32_MAX_DEVICES=4`)
- WiFi AP mode
- Log levels (INFO default, DEBUG max)
- Custom partition scheme (4 MB flash)

**DO NOT** check in `sdkconfig` (generated file, user-specific).

## PWM Frequency Tuning

### Current Default: 20 kHz @ 11-bit

**Rationale**:
- BTS7960 max: ~25 kHz
- Above audible range (no motor whine)
- 11-bit resolution = 2048 steps — sufficient for smooth DC motor control
- 12-bit at 20 kHz is borderline on ESP32 LEDC; 11-bit is the reliable sweet spot
- ESP32 LEDC actual output: ~19.5 kHz (80 MHz / 4096 with divider rounding)

### Changing PWM Frequency Safely

1. **Understand trade-off**: `frequency = 80MHz / (2^resolution)`
   - Higher freq → lower resolution
   - Lower freq → higher resolution (but may cause noise)

2. **Verify BTS7960 limits**: Don't exceed 25 kHz

3. **Update Kconfig**:
   - `CONFIG_ROBOT_MOTOR_PWM_FREQ_HZ` (frequency)
   - `CONFIG_ROBOT_MOTOR_PWM_RESOLUTION` (bit depth)

4. **Test incrementally**:
   - Start at 10 kHz, verify motor response
   - Increase to 15 kHz, 20 kHz, 25 kHz
   - Listen for audible noise
   - Check for overheating

5. **Document**: Update `docs/pwm-tuning.md` with test results

## CI/CD and Releases

### Workflow Overview

1. **ci.yml** (Firmware CI):
   - Triggers: Every push to `main` or `claude/**`, every PR to `main`
   - Installs ESP-IDF v5.1.2
   - Runs `idf.py build`
   - Uploads artifacts (bootloader, partition table, app binary)
   - **On every `main` push**: Creates/updates a rolling `latest` pre-release
     with the current binaries (so the web flasher always has real firmware)

2. **release.yml** (Release Build):
   - Triggers: Tag push (`v*.*.*`)
   - Builds firmware
   - Creates GitHub Release
   - Attaches binaries
   - Generates `manifest.json` for web flasher
   - Updates `latest.json` pointer

3. **pages.yml** (Web Flasher Deploy):
   - Triggers: Push to main, release creation
   - Builds web-flasher site
   - Deploys to GitHub Pages via `actions/deploy-pages`

### Creating a Release

```bash
git tag -a v1.0.0 -m "Release v1.0.0"
git push origin v1.0.0
```

CI will:
- Build firmware
- Create release with binaries
- Update web flasher manifest
- Deploy to GitHub Pages

**NEVER** manually create releases or upload binaries.

### Updating Web Flasher Manifest

**Automatic**: `release.yml` generates `manifest.json`:

```json
{
  "name": "Tracked Robot Firmware",
  "version": "1.0.0",
  "builds": [{
    "chipFamily": "ESP32",
    "parts": [
      {"path": "bootloader.bin", "offset": 4096},
      {"path": "partition-table.bin", "offset": 32768},
      {"path": "track-robot.bin", "offset": 65536}
    ]
  }]
}
```

If offsets change (new partition scheme), update `release.yml` script.

## Guardrails and Best Practices

### DO:
- Always read existing code before modifying
- Test code compiles before committing
- Update PROGRESS.md when completing tasks
- Document design decisions in `/docs`
- Use Kconfig for all configuration
- Log all state changes and errors
- Handle all error cases explicitly
- Keep functions focused and single-purpose
- Write self-documenting code with clear names

### DON'T:
- ✗ Break CI workflows (they must always pass)
- ✗ Change structure without updating this file
- ✗ Hardcode values (use Kconfig or #defines)
- ✗ Ignore compiler warnings
- ✗ Commit generated files (sdkconfig, build/)
- ✗ Skip error checking
- ✗ Add features not in scope (see Non-Goals)
- ✗ Remove safety features
- ✗ Change PWM frequency without testing
- ✗ Enable Bluedroid or esp_hidh (conflicts with Bluepad32/BTstack)

## Documentation Policy

### When to Document

**ALWAYS document**:
- Design decisions (why this approach?)
- API contracts (how to use this interface?)
- Configuration options (what do these do?)
- Safety considerations (what can go wrong?)
- Troubleshooting steps (how to fix common issues?)

**Update when**:
- Adding new control source → Add `/docs/myname-protocol.md`
- Changing architecture → Update `/docs/architecture.md`
- Changing workflows → Update `/docs/cicd.md`
- Completing milestone → Update `PROGRESS.md`

### Documentation Structure

Each doc file should have:
1. **Purpose**: What does this document cover?
2. **Context**: Why is this important?
3. **Details**: How does it work?
4. **Examples**: Show, don't just tell
5. **Troubleshooting**: Common issues and fixes

## Decision-Making Framework

When making a technical decision:

1. **Align with goals**: Does this serve a primary goal?
2. **Check non-goals**: Does this violate a non-goal?
3. **Consider safety**: Does this maintain or improve safety?
4. **Evaluate complexity**: Is this the simplest approach?
5. **Document**: Write down the decision and rationale

### Example Decision Log

```markdown
## Decision: Use Bluepad32 for Gamepad Input

**Context**: Need a reliable gamepad backend for PS4 DualShock 4.

**Options considered**:
- esp_hidh (ESP-IDF native): Unstable for gamepads; requires manual HID report parsing
- Bluepad32 + BTstack: Mature library, stable gamepad abstraction, active maintenance

**Decision**: Bluepad32 + BTstack

**Rationale**:
- Stable PS4/PS5/Xbox/Nintendo controller support out of the box
- No manual HID byte-offset parsing
- BTstack more reliable than Bluedroid for Classic BT HID gamepads

**Date**: 2026-05-08
**Documented in**: docs/architecture.md, docs/ps4-setup.md
```
